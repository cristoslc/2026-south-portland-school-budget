# Live Discovery over Static Config

## Context

The evidence pipeline connectors (Vimeo, budget page) use a config-driven model: a `--discover` flag enumerates available resources and writes them to a YAML config file (`vimeo-sources.yaml`, `budget-page-sources.yaml`). Subsequent runs download from that static list.

This design has a structural flaw exposed by BUG-001: bad or stale URLs persist in config and cause repeated failures. A Google Slides presentation URL using the `pubembed` format returns 404 on every run, causing the budget page connector to exit non-zero, which cascades to fail the entire pipeline workflow. The config file is an accumulation-only cache with no mechanism for removing or correcting entries.

More broadly, the config-driven model conflates "what's available" (a live, changing fact) with "what to download" (a derived decision). The YAML files serve no purpose beyond caching discovery results — they aren't curated, reviewed, or edited by humans.

## Decision

**Replace the config-driven connector model with live discovery on every run.**

Each connector invocation will:
1. **Enumerate** all available resources from the source (Vimeo channel, budget page)
2. **Filter** to FY27-relevant content (title prefixes, date ranges, document types)
3. **Diff** against what's already in the local repo (by filename or hash)
4. **Download** only missing resources

A shared **history file** (`data/<connector>/discovery.jsonl`) provides:
- Informative record of all discovered URLs for consumption by other processes
- Backoff support: URLs that fail are retried with exponential backoff (2^fail_count hours, capped at 48h)
- Not authoritative — connectors do not rely on the history to decide *what* to download, only *when* to retry failures

The static YAML config files (`vimeo-sources.yaml`, `budget-page-sources.yaml`) are removed.

## Alternatives Considered

**1. Fix the config-driven model (patch stale entries).**
Add URL validation, auto-removal of 404s, and config pruning. Rejected because it treats symptoms — the fundamental issue is that a static file cannot represent a dynamic source. Every new failure mode would need its own patch.

**2. Hybrid: config for curation, live discovery for additions.**
Keep the config but make discovery append-only with human review. Rejected because no one reviews the config — it's generated by automation and consumed by automation. The curation layer adds complexity without value.

**3. Live discovery with no history file.**
Enumerate and download every run with no memory of past failures. Rejected because at hourly frequency, a persistent 404 would generate noise in logs and waste time on every run. The history file with backoff is low-cost and high-value.

## Consequences

**Positive:**
- Eliminates the entire "stale config" failure class (resolves BUG-001)
- Connectors are simpler — no config read/write machinery
- New content is picked up automatically with zero manual intervention
- History file supports backoff, reducing noise from persistent failures
- History file is consumable by other tools (dashboards, audits) without running discovery

**Negative:**
- Every run hits the source APIs (one `yt-dlp --flat-playlist` call, one HTTP fetch + parse). At hourly frequency this is negligible but worth monitoring.
- Existing SPEC-009 and SPEC-010 are superseded — their implementations will be replaced
- Slightly more complex filter logic in connector code (previously implicit in which URLs were in config)
