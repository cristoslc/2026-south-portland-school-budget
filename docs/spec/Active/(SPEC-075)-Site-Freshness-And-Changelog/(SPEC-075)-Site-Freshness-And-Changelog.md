# SPEC-075: Site Freshness Indicators and Changelog

## Problem

Visitors have no way to know how current the information is. Briefings show `generated_date` in metadata but there's no site-wide "last refreshed" indicator, and no changelog showing what changed between updates.

## Design

### 1. Per-page "Last refreshed" footer line

Add a line above the existing site footer on content pages:

- **Briefing pages:** "Last refreshed: {generated_date}"
- **Persona pages:** "Last updated: {last-updated}"
- **Home page:** "Briefings last refreshed: {most recent generated_date across all briefings}"

Use the existing `.briefing-meta` styling pattern. Subtle, not prominent.

### 2. Changelog sidecar

A markdown file at `dist/changelog.md` generated by `track2-cron.sh` (or a dedicated script) after each successful pipeline run. Format:

```markdown
# Site Updates

## 2026-03-24
- Regenerated 17 briefings for upcoming 2026-03-23 meeting
- Added cumulative folds for 2026-03-10-city-council (15 personas)
- New evidence: FY27 Budget presentation and budget book (2026-03-23)

## 2026-03-23
- Regenerated 17 briefings for upcoming 2026-03-23 meeting
- Added cumulative folds for 2026-03-05-city-council (15 personas)
```

### 3. Changelog page on site

New page at `/changelog/` that renders `dist/changelog.md`. Add to site nav (secondary, footer link — not main nav).

## Acceptance Criteria

- [ ] Briefing detail pages show "Last refreshed: {date}" near bottom
- [ ] Persona detail pages show "Last updated: {date}" near bottom
- [ ] Home page shows most recent briefing refresh date
- [ ] `dist/changelog.md` exists and is populated
- [ ] `/changelog/` page renders on the site
- [ ] Footer links to changelog
